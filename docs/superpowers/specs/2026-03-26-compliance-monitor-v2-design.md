# Compliance Monitor V2 ‚Äî Auth & Collaboration Design

## Goal

Create a new copy of the compliance monitor project with a login wall, user collaboration features (favorites, article sharing, comment threads, direct messages, notifications, user submissions), and a clean modular code foundation. The existing production project is untouched throughout development.

## Architecture

**Infrastructure stack:**
- **Vercel** ‚Äî replaces GitHub Pages. Serves the same static HTML files generated by the Python pipeline, but behind an auth wall. Free tier. Auto-deploys from GitHub on every push, same as GitHub Pages does today.
- **Supabase** ‚Äî new addition. Handles auth, user management, and all user-specific data (favorites, comments, messages, notifications). Free tier, upgradeable to Pro ($25/month) when needed.
- **SQLite** ‚Äî unchanged. All regulatory pipeline data (articles, deadlines, bills, topic insights) stays in the local SQLite database. The pipeline writes to SQLite and pushes rendered HTML to GitHub exactly as it does today.
- **Python pipeline** ‚Äî unchanged. No modifications to run.py, summarizer.py, scrapers, or any pipeline code.
- **Cloudflare Worker** ‚Äî unchanged. API proxy for the deep-dive viewer stays as-is.

**Key architectural principle:** Supabase handles only user-specific runtime data. The dashboard content is still pre-rendered static HTML. Supabase JS client is added to page templates to layer user features on top without changing how the pages are built or served.

**Multi-tenancy note:** If this ever becomes a multi-company SaaS product, one column (`company_id`) gets added to the `profiles` table and Supabase row-level security handles data isolation automatically. No other schema changes needed. Design for this by keeping all user data keyed to `user_id` ‚Äî no assumptions about single-company use.

## Tech Stack

Python 3, Jinja2, SQLite, Supabase (Postgres + Auth + Realtime), Vercel, Supabase JS client (`@supabase/supabase-js` via CDN), existing CSS/JS patterns.

## Step 0 ‚Äî Modular Template Refactor

Before any auth or feature work, the new repo gets a clean foundation. This is done once at the start and makes every subsequent step easier.

**What changes:**
- Extract shared CSS from all dashboard templates into `dashboard/static/css/dashboard.css`. Every page loads this external stylesheet instead of embedding thousands of lines of inline CSS.
- Split JavaScript into feature files in `dashboard/static/js/`: `nav.js` (sidebar accordion, resize handles), `filters.js` (article filtering, search), `auth.js` (Supabase auth checks), `notifications.js` (bell feed), `favorites.js` (save/share articles), `comments.js` (comment threads), `messages.js` (DMs).
- Break `dashboard.html` into Jinja2 template partials: `_topbar.html`, `_sidebar.html`, `_article_card.html`, `_badge_legend.html`. Pages use `{% include %}` to compose from these.
- **Email templates are excluded.** `base.html`, `topic_section.html`, and `exec_summary.html` keep inline CSS because `premailer` requires it for email client compatibility.

**What stays the same:** All visual design, layout, colors, badge logic, functionality. This is a structural refactor only ‚Äî nothing visible changes.

## Step 1 ‚Äî Infrastructure Setup

1. Create a new GitHub repository as a copy of the current project at its current state.
2. Create a Supabase project. Note the project URL and anon key for `.env`.
3. Create a Vercel project connected to the new GitHub repo. Configure it to serve from the repo root (the generated HTML files are pushed directly to the repo by the pipeline).
4. Update the pipeline's GitHub push target to the new repo.
5. Add Supabase credentials to `.env`: `SUPABASE_URL`, `SUPABASE_ANON_KEY`.
6. Add `.superpowers/` to `.gitignore`.

**DNS:** Vercel provides a deployment URL immediately. GitHub Pages URL is left running on the production project ‚Äî no DNS changes to the production site.

## Step 2 ‚Äî Auth & User Management

### Login wall

Every page checks for a valid Supabase session on load via `auth.js`. If no session exists, the user is redirected to `/login.html` immediately. No content is rendered without a session ‚Äî Vercel serves the HTML but `auth.js` replaces the page body with the redirect before anything renders.

`/login.html` is a standalone page with two options:
- **Sign in** ‚Äî email + password
- **Request access** ‚Äî name + work email, submits to `access_requests` table

### Access request flow

1. Requester fills out name + email on `/login.html` and submits.
2. A row is inserted into `access_requests` with `status = 'pending'`.
3. A Supabase Edge Function fires and sends an email to the super admin (and any admins) with approve/reject links.
4. Admin clicks approve ‚Üí Edge Function creates the Supabase Auth user, sets `is_approved = true` on their `profiles` row, updates `access_requests` status.
5. New user receives an email with a link to set their password and log in.
6. Admin clicks reject ‚Üí `access_requests` status set to `rejected`, optional rejection email sent to requester.

### User roles

Three roles stored in `profiles.role`:
- `super_admin` ‚Äî full access, cannot be demoted or deleted by anyone else
- `admin` ‚Äî can approve/reject access requests, view all users, promote users to admin, deactivate users. Cannot see other users' private favorites. Cannot demote super_admin.
- `user` ‚Äî standard access to all dashboard content and collaboration features

### Admin panel (`/admin.html`)

Accessible only to `admin` and `super_admin` roles. Shows:
- Pending access requests (approve / reject buttons)
- Active users table (name, email, role, last seen, deactivate button)
- Promote user to admin button

## Step 3 ‚Äî Favorites & Article Tagging

### Saving an article

Every article card in the article list gets a bookmark icon (Lucide `Bookmark`) on hover. Clicking it:
- If not yet saved: inserts a row into `saved_articles`, icon fills in
- If already saved: deletes the row, icon unfills

The `article_title`, `article_url`, and `article_topic` are stored in Supabase at save time (denormalized from the rendered page) so favorites can be displayed without querying SQLite.

### Sending an article to a colleague

A second icon on each article card (Lucide `UserPlus`) opens a small inline panel:
- Dropdown to select a team member from `profiles`
- Optional text field for a note
- Send button

On send:
1. Row inserted into `article_tags`
2. Row inserted into `notifications` for the recipient (`type = 'article_tag'`)
3. Supabase Edge Function sends an email to the recipient with the article title, sender name, note, and a link directly to the article

### My Saved Articles page (`/saved.html`)

A dedicated page showing all of the logged-in user's saved articles, grouped by topic, with the date saved. Articles can be unsaved from this page.

## Step 4 ‚Äî Comment Threads

Comment threads appear on:
- Individual articles (in the article card, expanded on click)
- Deadline cards
- Legislative bill cards

### Comment structure

A "Comments" toggle button on each supported card. Clicking it expands a thread panel below the card. The thread shows existing comments and a text input to add a new one.

Top-level comments have `parent_id = null`. Reply buttons on each comment create a new comment with `parent_id` set to the parent's `id`. Threads are limited to two levels (comment + reply) to keep the UI clean.

Each comment shows: author name, timestamp, body text. Comment authors can edit or delete their own comments (soft delete ‚Äî body replaced with "[deleted]", record kept for thread integrity).

### Notifications for comments

When someone replies to your comment, a `notifications` row is inserted (`type = 'comment_reply'`) and an email is sent.

## Step 5 ‚Äî Direct Messages

### Accessing DMs

A message icon (Lucide `MessageSquare`) in the topbar alongside the notification bell. Clicking opens `/messages.html`.

### Messages page (`/messages.html`)

Left column: list of conversations (one per person you've messaged), sorted by most recent. Shows the other person's name and a preview of the last message.

Right column: the active conversation thread. Messages appear in chronological order. Text input at the bottom to send a new message.

### Real-time delivery

Supabase Realtime subscriptions push new messages to the browser instantly ‚Äî no polling or page refresh. The message icon in the topbar shows an unread count badge when new DMs arrive.

### Article attachments

An optional "Attach article" button in the compose area lets the sender link an article from the current page context. The attachment renders as a linked card in the message thread.

## Step 6 ‚Äî Notifications

### Bell feed

A bell icon (Lucide `Bell`) in the topbar with a red badge showing the count of unread notifications. Clicking opens a dropdown panel showing the 10 most recent notifications.

Each notification shows:
- Icon indicating type (tag, comment reply, DM, access approved)
- Short title ("Ryan tagged you in a PFAS article")
- Relative timestamp ("2 hours ago")
- Clicking the notification navigates to the relevant content and marks it as read
- "Mark all as read" button at the top of the dropdown

### Notification types

| Type | Trigger | Email sent? |
|---|---|---|
| `article_tag` | Someone sends you an article with a note | Yes |
| `comment_reply` | Someone replies to your comment | Yes |
| `dm` | New direct message received | Yes (digest ‚Äî not per message) |
| `access_approved` | Your access request was approved | Yes |

### Email notifications

Sent via Supabase Edge Functions using Supabase's built-in SMTP relay (configured in the Supabase dashboard ‚Äî no separate email service needed). Emails include the notification content and a direct link. DM emails are batched ‚Äî a single "You have new messages" email rather than one per message.

### Row-level security rules (all tables)
- `profiles` ‚Äî users can read all profiles (needed for @mentions, DM recipient picker), can only update their own row
- `access_requests` ‚Äî readable and writable by admins only; insert by anyone (unauthenticated) for the request form
- `saved_articles` ‚Äî users see and modify only their own rows
- `article_tags` ‚Äî visible to sender and recipient only
- `comments` ‚Äî readable by all authenticated users; writable by author only (edits/deletes)
- `messages` ‚Äî visible to sender and recipient only
- `notifications` ‚Äî users see only their own rows
- `user_submissions` ‚Äî readable by all authenticated users; writable by submitter only (edits/deletes)

## Step 7 ‚Äî User Submissions

Team members can submit content they find outside the automated pipeline: external URLs and PDF documents. This ensures things the scrapers miss can still enter the system and participate in the full collaboration layer.

### URL submissions

A "Submit Article" button appears in the topbar and on each topic page. Clicking opens a small form:
- URL (required)
- Topic (dropdown: PFAS, EPR, REACH, TSCA, Prop65, ConflictMinerals, ForcedLabor)
- Context note (optional ‚Äî "Found this on the EPA site, relevant to our MN suppliers")
- Submit button

On submit, the system attempts to scrape the page title and a short snippet from the URL (via a Supabase Edge Function). If scraping fails, the user's own title field is used as a fallback. The submission is stored in `user_submissions` and immediately visible to the team.

**Where it appears:**
- On the relevant topic page: in the main article feed, mixed with pipeline articles, with a "Submitted by [name]" badge to distinguish it from pipeline-scraped content
- On a dedicated `/submissions.html` page showing all team submissions across topics

### Document submissions (PDFs)

For documents (PDFs, occasional Word files), a "Submit Document" option on the same form allows file upload. Files are stored in Supabase Storage. Documents do not appear in the main pipeline article feed ‚Äî they appear only in the Team Submissions section on the relevant topic page and on `/submissions.html`.

### Collaboration on submissions

Both URL submissions and documents support the full collaboration layer:
- Save to favorites
- Share to a colleague with a note (creates `article_tags` row using the submission's URL/path as `article_url`)
- Comment threads (uses `comments` table with `target_type = 'submission'`)

### Project linking (forward-compatible)

Every piece of content in the system ‚Äî pipeline articles, user-submitted URLs, documents, deadlines, bills ‚Äî displays two action buttons that are implemented as stubs in this phase:

- **"Start Project"** ‚Äî creates a new compliance project with this item pre-attached (implemented in the project management spec)
- **"Add to Project"** ‚Äî opens a picker to attach this item to an existing project (implemented in the project management spec)

In this phase, both buttons are visible but display a "Coming soon" tooltip when clicked. The buttons are built now so the UI pattern is consistent everywhere before the project management system exists. This means no awkward retrofit when project management is built.
